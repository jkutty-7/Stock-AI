### ALPHA-2: Adaptive Confidence Scoring (Feedback Loop)

**What**: After every closed outcome, recalibrate the confidence model. Bucket historical signals by confidence range (0.5-0.6, 0.6-0.7, etc.) and compare assigned confidence vs actual win rate. Adjust future signals accordingly.

**Why**: This is the single most important intelligence feature. Without it, the system repeats the same mistakes. If Claude assigns 0.8 confidence to signals that win only 50%, the system learns to discount those. If 0.55-confidence signals actually win 70%, the system trusts them more. This is how you go from 55% to 65% win rate over 6 months.

#### How It Works

```
1. Outcome tracker closes a signal → triggers calibration (debounced, max once/hour)
2. Calibrator groups all closed outcomes into confidence buckets
3. For each bucket: actual_win_rate vs bucket_midpoint
4. Adjustment formula: adjusted = raw_confidence × (actual_win_rate / bucket_midpoint)
5. Clamped to [0.30, 0.95]
6. Segmented by: action_type, strategy_tag, time_horizon, regime
```

**Example**: If the 0.7-0.8 bucket has 45% actual win rate (midpoint 0.75):
- Adjustment factor = 0.45 / 0.75 = 0.60
- A new 0.75 confidence signal becomes 0.75 × 0.60 = 0.45 (downgraded)

### ALPHA-3: Tiered Risk Management

**What**: Replace the binary 8% drawdown breaker with a graduated 4-tier system. Add volatility-adjusted position sizing and sector concentration monitoring.

**Why**: The current system goes from "full throttle" to "full stop" at 8%. Professional risk management uses gradual reduction. A 3% drawdown should reduce exposure — not wait until 8% when recovery is much harder.

#### Risk Tiers

| Tier | Drawdown | Position Size Multiplier | Min Confidence | Max Exposure |
|------|----------|-------------------------|----------------|--------------|
| **GREEN** | 0-3% | 1.0x (full) | 0.60 | 90% |
| **YELLOW** | 3-5% | 0.7x | 0.70 | 75% |
| **ORANGE** | 5-8% | 0.5x | 0.80 | 50% |
| **RED** | >8% | 0.0x (halted) | N/A | 20% |

#### Volatility-Adjusted Position Sizing

```
adjusted_qty = base_qty × (avg_atr_20d / current_atr)
```

When volatility doubles → position size halves. When volatility is below average → position size increases. This prevents large losses during volatile periods and maximizes exposure during calm trends.

#### Sector Heat Mapping

- Track sector concentration: if >40% of portfolio value in one sector → WARNING alert
- Track correlation: if >2 positions are in the same sector → flag in Claude prompt context
- Prevent new entries in over-concentrated sectors (configurable)

### ALPHA-4: Session-Aware Trading Bias

**What**: Segment the trading day into 4 sessions with different entry criteria, stop-loss multipliers, and volume requirements.

**Why**: 60-70% of NSE intraday volume occurs in the first and last hours. Noon trades have the worst win rates across all markets. Knowing when NOT to trade is as valuable as knowing when to trade.

#### Session Definitions

| Session | Time (IST) | Behavior | SL Multiplier | Min Confidence |
|---------|-----------|----------|---------------|----------------|
| **First Hour** | 9:15 - 10:15 | Aggressive: gap trades, ORB. High volatility expected. | 1.5x (wider) | 0.55 |
| **Mid-Morning** | 10:15 - 12:00 | Normal: trend continuation trades. Standard parameters. | 1.0x | 0.65 |
| **Lunch** | 12:00 - 13:30 | Defensive: low volume, choppy. Avoid new entries unless exceptional. | 0.8x (tighter) | 0.85 |
| **Closing Hour** | 14:00 - 15:15 | Selective: position squaring, mean-reversion setups only. | 1.0x | 0.75 |

#### Gate Function

```python
def should_enter_now(trigger_type, confidence, current_session):
    """Returns True if entry is allowed in current session."""
    if session == "LUNCH" and confidence < 0.85:
        return False  # Skip low-conviction lunch entries
    if session == "CLOSING" and trigger_type == "ORB_BREAKOUT":
        return False  # ORB breakouts don't work in closing hour
    ...
```

### ALPHA-5: Partial Profit-Taking (Scaled Exits)

**What**: Replace all-or-nothing exits with a 3-tier exit strategy. Exit portions at different targets, trailing the remainder for larger wins.

**Why**: The current system exits 100% at target — maximum reward is capped at ~1.5x risk. With partial exits, average winners become 1.8-2.2x risk while losers remain at 1x. This dramatically improves the profit factor.

#### Exit Tiers

| Tier | % of Position | Target | After Fill |
|------|--------------|--------|------------|
| **T1** | 50% | 1:1 risk-reward | Move SL to breakeven |
| **T2** | 30% | 1:2 risk-reward | Trail SL at T1 level |
| **T3** | 20% | Trailing 1.5× ATR | Let the winner run |

#### Example

Entry: ₹100, SL: ₹98 (risk = ₹2), Quantity: 100 shares

| Event | Action | Remaining | New SL |
|-------|--------|-----------|--------|
| Price hits ₹102 (T1) | Sell 50 shares, lock ₹100 profit | 50 shares | → ₹100 (breakeven) |
| Price hits ₹104 (T2) | Sell 30 shares, lock ₹120 profit | 20 shares | → ₹102 (T1 level) |
| Price hits ₹108, trails back | T3 trailing SL triggers at ₹106.5 | 0 shares | — |
| **Total P&L** | 50×₹2 + 30×₹4 + 20×₹6.5 = **₹350** vs ₹200 with all-at-target | | |

### BETA-1: Modular Strategy Framework

**What**: Extract the current hardcoded strategies (ORB breakout, VWAP cross, Supertrend flip) into a pluggable strategy registry. Each strategy is a class with a standard interface. Add two new strategies: Mean Reversion and Momentum Breakout.

**Why**: The current code mixes strategy logic into `intraday_monitor.py`. Adding a new strategy means modifying that method. A framework enables: per-strategy performance tracking, data-driven strategy selection, A/B testing, and the weekly self-review (V3-1) to enable/disable strategies based on performance.

#### Strategy Interface

```python
class BaseStrategy(ABC):
    name: str
    timeframe: str                           # "intraday" | "swing" | "positional"
    required_indicators: list[str]

    @abstractmethod
    def evaluate(self, data: StrategyInput) -> Optional[StrategySignal]:
        """Return a signal if conditions are met, None otherwise."""

    @abstractmethod
    def get_exit_levels(self, entry_price: float, data: StrategyInput) -> dict:
        """Return {stop_loss, t1, t2, t3} price levels."""
```

#### Strategies

| Strategy | Type | Entry Condition | Edge |
|----------|------|-----------------|------|
| `ORBBreakout` | Intraday | Price > ORB high + 0.1%, 3 confirm ticks | Existing, extracted |
| `VWAPCross` | Intraday | Price crosses VWAP up, ≥3 consecutive up ticks | Existing, extracted |
| `SupertrendFlip` | Intraday | Direction flips to UP on 5-min candles | Existing, extracted |
| **`MeanReversion`** | Intraday/Swing | RSI < 30 + price at support + bullish divergence + session ≠ lunch | **New** — high-alpha reversal setup |
| **`MomentumBreakout`** | Intraday/Swing | ADX > 25 + price breaks 20-day high + volume > 2x avg | **New** — works any time of day |

#### Strategy Registry

```python
class StrategyRegistry:
    def __init__(self):
        self.strategies: dict[str, BaseStrategy] = {}

    def register(self, strategy: BaseStrategy): ...

    def evaluate_all(self, data: StrategyInput) -> list[StrategySignal]:
        """Run all enabled strategies, return signals sorted by confidence."""

    def get_performance(self, strategy_name, days=30) -> dict:
        """Win rate, avg P&L, trade count for a strategy."""
```

The registry reads `enabled_strategies` from config and `strategy_weights` from MongoDB (set by weekly review in V3-1).

### BETA-2: Backtesting Engine

**What**: Replay historical candle data through the strategy framework. Compute win rate, profit factor, max drawdown, Sharpe ratio, Sortino ratio, and maximum underwater duration. Output per-strategy and combined reports.

**Why**: Without backtesting, you cannot know if a strategy has positive expectancy before risking real capital. This is non-negotiable for any serious trading system. The backtester runs Python strategies only — zero Claude API cost.

#### Core Engine

```python
class Backtester:
    def run(
        self,
        strategy: BaseStrategy,
        symbols: list[str],
        start_date: date,
        end_date: date,
        initial_capital: float = 100000.0
    ) -> BacktestResult:
        """
        1. Fetch historical candles from MongoDB (or Groww API fallback)
        2. For each candle: compute indicators, evaluate strategy
        3. Track virtual positions with entry/exit/partial exits
        4. Compute equity curve and performance metrics
        """
```

#### Performance Metrics

| Metric | Formula | What It Tells You |
|--------|---------|-------------------|
| **Win Rate** | wins / total_trades | Hit rate |
| **Profit Factor** | gross_profit / gross_loss | >1.5 is good, >2.0 is excellent |
| **Max Drawdown** | max(peak - trough) / peak | Worst-case capital loss |
| **Sharpe Ratio** | mean(daily_returns) / std(daily_returns) × √252 | Risk-adjusted return (>1.0 is good) |
| **Sortino Ratio** | mean(daily_returns) / std(negative_returns) × √252 | Downside risk-adjusted (better than Sharpe) |
| **Max Underwater Days** | Longest period below equity peak | Recovery resilience |
| **Expectancy** | (win_rate × avg_win) - (loss_rate × avg_loss) | Per-trade expected profit |

#### Interfaces

- **Telegram**: `/backtest STRATEGY DAYS` (e.g., `/backtest orb_breakout 90`)
- **REST API**: `POST /api/v1/backtest` with `{strategy, symbols, start_date, end_date}`
- **Claude Tool**: `run_backtest` — Claude can backtest when user asks

### V3-1: Self-Improving Weekly Review

```
Sunday 8:00 PM IST:
1. Pull all closed outcomes for the week
2. Group by: strategy × regime × session × symbol_sector

ADJUSTMENTS:
- Strategy win rate < 40% for 4 weeks → DISABLE strategy
- Strategy win rate > 70% for 2 weeks → INCREASE weight by 1.2x
- Regime+strategy combo underperforms → ADD constraint (higher min confidence)

- Strategy registry reads these on each evaluation cycle
- Send formatted report to Telegram
```

### V3-2: Signal Staleness & Time Decay

**What**: Signals that are not acted upon within a configurable window automatically decay in confidence and eventually expire. A BUY signal at 10:00 AM with 0.8 confidence is not equally valid at 2:00 PM — market conditions have changed.

**Why**: The current system generates signals that persist until manually closed. Stale signals create false confidence and lead to entries based on outdated analysis. Time decay forces the system to re-evaluate or discard old signals.

#### Decay Model

```
Exponential decay: confidence(t) = original × 2^(-t / half_life)

Half-lives by time horizon:
  Intraday:    30 minutes
  Swing (3-5d): 24 hours (1440 minutes)
  Positional:   72 hours (4320 minutes)

Expiration: When decayed confidence < 0.30, mark as EXPIRED
```

#### Example

| Time | Original | Decayed (intraday, 30m half-life) |
|------|----------|-----------------------------------|
| +0 min | 0.80 | 0.80 |
| +15 min | 0.80 | 0.67 |
| +30 min | 0.80 | 0.56 |
| +45 min | 0.80 | 0.47 |
| +60 min | 0.80 | 0.40 |
| +90 min | 0.80 | 0.28 → EXPIRED |

### V3-3: Redis Caching Layer

**What**: Replace the in-memory `TTLCache` with Redis for cross-process caching, configurable TTLs, and cache-aside pattern for expensive computations (indicator calculations, screener results, enriched quotes).

**Why**: The current `TTLCache` is per-process and lost on restart. Redis enables: caching indicator results across monitoring cycles (saving Groww API calls), sharing state between FastAPI and background tasks, and faster warm-up after restart. For 500+ stock screener runs, this saves hundreds of redundant API calls.

#### Cache Tiers

| Data | TTL | Rationale |
|------|-----|-----------|
| LTP quotes | 8s | Real-time, must be fresh (unchanged from v2.2) |
| OHLCV candles (intraday) | 60s | 1-min candles, moderate freshness |
| Technical indicators (daily) | 5 min | Daily indicators don't change within 5 min |
| Enriched quotes | 5 min | Indicator computation is expensive |
| Screener results | 1 hour | Screener runs once daily, results stable |
| Regime classification | 6 hours | Computed once daily, very stable |

#### Graceful Degradation

```python
class SmartCache:
    def __init__(self, redis_url: str = ""):
        if redis_url:
            self._backend = RedisCache(redis_url)
        else:
            self._backend = TTLCache()  # Fall back to in-memory
```

If Redis goes down mid-session, automatically fall back to in-memory cache with a WARNING log. No data loss, no service interruption.

## Deliberately Excluded

These features were evaluated and intentionally deferred to v3.5 or beyond:

| Feature | Reason for Exclusion |
|---------|---------------------|
| **Options chain / IV / Greeks** | Groww API does not expose options chain data for retail accounts. Would require a separate data provider (NSE Bhavshare or paid feeds). Defer to v3.5 when broker migration or paid data integration is considered. |
| **FII/DII institutional flow data** | Only available via NSE website scraping (fragile) or paid vendors. Data is lagging (published T+1). Alpha-to-effort ratio is low for the current system maturity. |
| **News/sentiment analysis** | Requires paid news API (Bloomberg, Reuters) or web scraping + NLP pipeline. High complexity, moderate alpha. Better suited after the core intelligence loop is solid. |
| **WebSocket real-time streaming** | Groww does not offer a WebSocket API. 10-second polling is near their rate limit. No practical upgrade path without changing brokers. |
| **Automated order execution** | The current system generates signals but does not place orders. This is deliberate — regulatory and risk reasons. The Telegram alert + manual execution model is safer while the system builds a track record. Revisit after 6 months of positive outcome tracking. |
| **Ensemble AI models** | Running multiple Claude models would multiply API costs by N. The self-improving feedback loop (ALPHA-2 + V3-1) achieves similar accuracy improvement at 1x cost. |
| **Tick-level data (sub-second)** | Not available from Groww. Would require co-located market data feed. Overkill for the current strategy set. |
| **Multi-broker support** | Single broker (Groww) is sufficient for current scale. Abstraction layer adds complexity without immediate benefit. |

## Summary

v3.0 transforms Stock-AI from a signal generator into a self-improving trading system:

| Capability | v2.2 | v3.0 |
|-----------|------|------|
| **Indicators** | 8 standard + 4 intraday | 15+ standard + 6 intraday + candle patterns + S/R + Fibonacci |
| **Timeframes** | Single (daily or 5-min) | Multi-timeframe (5m/1H/4H/daily) with alignment scoring |
| **Strategies** | 3 hardcoded (ORB/VWAP/ST) | 5+ pluggable via registry, with performance tracking |
| **Confidence** | Static (Claude assigns, never adjusted) | Adaptive (calibrated from actual outcomes, decays over time) |
| **Risk** | Binary (8% on/off) | 4-tier graduated + volatility-adjusted sizing + sector heat map |
| **Exits** | All-or-nothing | 3-tier partial exits (T1/T2/T3) |
| **Session awareness** | None | 4 sessions with different rules |
| **Backtesting** | None | Full replay engine with Sharpe/Sortino/profit factor |
| **Self-improvement** | None | Weekly autonomous review, strategy enable/disable, weight adjustment |
| **Signal lifecycle** | Persist forever | Exponential decay, auto-expiration |
| **Performance monitoring** | Basic win rate | Full equity curve, Sharpe/Sortino, degradation detection |
| **Caching** | In-memory (lost on restart) | Redis with graceful fallback |
